# 兰台清晖界面改版设计

日期：2026-03-26

## 1. 背景与目标

当前项目的登录页与内部工作台已经具备完整业务能力，但品牌名称、视觉气质与页面层次仍偏通用后台风格，难以体现“等保测评中的定级备案过程”所要求的审慎、秩序和正式感。

本次改版目标：

- 将平台主品牌从“定级备案管理系统”升级为更有识别度的古典命名。
- 在不改变业务流程和接口行为的前提下，提升登录页和内部页面的视觉完成度。
- 让页面既有文化气质，又保持政企工具型系统应有的清晰、可信和易用。

## 2. 设计结论

### 2.1 品牌命名

- 主品牌：`兰台清晖`
- 功能副标：`等保测评定级备案工作台`

命名理由：

- `兰台`带有典籍、档案、文书和制度归档的古典意味，贴合“备案、留痕、管理”语境。
- `清晖`补足清朗、严整、克制的气质，避免名称过于板滞。
- 主品牌负责识别度，副标负责业务直达，避免用户只看到雅称却不知用途。

### 2.2 登录页题签文案

- 主标题：`兰台清晖`
- 引句：`物有本末，事有终始。知所先后，则近道矣。`
- 功能说明：`用于等保测评中的定级研判、材料归档与备案流程协同`

设计原则：

- 名字最大，作为品牌锚点。
- 引句次之，作为气质表达，不喧宾夺主。
- 功能说明最克制，负责把业务说清楚。

## 3. 页面类型与语言策略

- 页面类型：工具型后台界面，局部加入品牌化表达。
- 主要语言：简体中文。
- 文案策略：导航、按钮、提示语仍保持功能导向；品牌区、题签区、少量说明文案承接古典气质。

不采用的方案：

- 不把整个系统写成“古风文案站点”。
- 不把所有导航和模块名替换成抽象雅称。
- 不加入与业务无关的大段诗词或装饰性宣言。

## 4. 改造范围

本次仅改视觉与文案层，不改业务能力。

明确范围：

- 登录页：`app/templates/auth.html`
- 基础框架：`app/templates/base.html`
- 全局样式：`app/static/style.css`
- 必要的模板测试：`tests/test_ui_templates.py`

明确不改：

- FastAPI 路由与接口行为
- 表单字段 ID、JS 逻辑入口与请求参数
- 登录、改密、鉴权、导航与角色控制逻辑
- 业务模块的核心交互路径

## 5. 视觉方向

### 5.1 总体气质

总体方向为“典雅克制，局部见锋芒”。

- 以正式文书感替代常见互联网后台风格。
- 以层次、留白、边界和材质感体现设计完成度。
- 炫技点放在版式、纹理、题签、过渡和细节，而不是高饱和渐变和夸张动效。

### 5.2 色彩系统

主色系：

- 黛青 / 墨蓝：作为品牌主色和重点背景
- 绢白 / 宣纸白：作为大面积内容背景
- 竹灰 / 石青灰：作为边框、弱化文字和分隔

点缀色：

- 朱砂红：少量用于印记感、重要状态或强调点
- 铜金棕：少量用于品牌细节、题签边线或装饰点

约束：

- 不使用“蓝紫科技光效”作为主视觉语言。
- 不大面积使用红金，避免视觉风格滑向节庆化或传统海报化。

### 5.3 字体策略

- 品牌标题与引句使用偏宋韵的中文衬线回退栈，优先营造“题签/文书”气质。
- 正文、表单、表格、按钮继续使用清晰的中文无衬线字体。

建议实现时直接落成以下回退栈：

- 品牌标题 / 引句：`"STSong", "SimSun", "Noto Serif CJK SC", "Source Han Serif SC", serif`
- 正文 / 表单 / 表格：`"PingFang SC", "Microsoft YaHei", "Noto Sans CJK SC", "Source Han Sans SC", sans-serif`

目标：

- 品牌有辨识度。
- 业务区保持高可读性和稳定密度。

### 5.4 材质与背景

- 背景加入极轻的纸纹、题签纹理或墨色层次。
- 面板保留现代卡片结构，但通过边框、角度、阴影和局部纹理形成“笺纸面板”质感。
- 视觉表达应服务结构，不能影响表格可读性和输入控件对比度。

### 5.5 基础设计 Tokens

实现阶段应至少明确以下基础 tokens：

- 主品牌色：黛青 / 墨蓝
- 页面底色：绢白 / 宣纸白
- 文本层级：主文本、次文本、弱化文本
- 语义色：成功、警示、错误
- 边框层级：常规边框、强调边框、弱分隔
- 圆角层级：大面板、中面板、控件
- 阴影层级：悬浮面板、常规面板、轻量卡片
- 间距层级：页面外边距、面板内边距、控件间距
- 动效时长：常规过渡、浮层进出、路由切换

这些 tokens 可以在 CSS 变量中落地，不要求额外引入设计系统框架。

## 6. 页面结构设计

### 6.1 登录页

结构保持左右双栏，但重做品牌区层级。

左侧品牌区：

- 品牌标识与平台名：`兰台清晖`
- 引句题签：`物有本末，事有终始。知所先后，则近道矣。`
- 功能说明：说明平台用于等保测评中的定级备案协同
- 三个业务特征点：围绕定级要素、材料留痕、流程节点，不使用泛安全口号

右侧操作区：

- 保留单一登录表单
- 保留首次登录强制改密的原流程
- 保留“清除当前登录”等现有操作
- 优化标题、说明、状态提示和按钮层级

约束：

- 不引入多 tab 登录
- 不增加新的登录流程
- 不改现有 JS 函数名和调用关系

### 6.2 内部工作台

保留现有导航与内容骨架，但统一品牌化表达。

侧边栏：

- 品牌区改为更具题签感的布局
- 名称替换为 `兰台清晖`
- 副标展示为 `等保测评定级备案工作台`

顶部区域：

- 保留“当前模块 + 当前账号状态 + 登录/退出入口”的结构
- 视觉上从通用后台头部提升为“模块题头”

页面 Hero：

- 保留 `page-hero` 结构
- 弱化营销感，强化“卷宗说明 / 模块说明”的稳定感
- 文案保持直白清晰，不采用过度修辞

页面变体：

- `standard` 页面继续保留 `page-hero`
- `management-tight` 页面继续保持紧凑模式，不新增 `page-hero`
- 紧凑模式主要用于模板、知识库、用户管理、备份恢复等管理页，改动重点放在顶部、面板和内容密度，而不是再加一层头图区

内容区：

- 表单、表格、卡片统一风格
- 重点通过留白、边框、标题、状态标识和局部材质统一视觉语言

## 7. 关键组件策略

### 7.0 共享组件清单

本次改版优先统一以下共享界面元素：

- 侧边栏品牌区
- 顶部模块题头
- `page-hero`
- `panel` 容器
- 表单输入区与分组标题
- 表格容器与表头
- 状态提示 / 结果提示容器
- 登录页品牌区与登录操作区

### 7.1 导航

- 功能命名保持现有直白中文，如“系统信息”“报告与审核”“流程管控”
- 不把古典命名塞进业务导航，避免理解门槛上升

### 7.2 卡片

- 减少互联网数据看板风格
- 通过颜色克制、边界清晰和题签化细节提升质感
- 不新增伪数据或装饰性指标

### 7.3 表格

- 保持高信息密度和清晰对比
- 背景和斑马纹轻量处理
- 避免重纹理压住正文信息

### 7.4 表单

- 保持现代输入控件体验
- 焦点态、错误态和悬停态清晰可辨
- 可在标签、分组标题和说明文案上体现品牌风格

### 7.5 状态与提示

- 状态色仍采用现代可识别语义色
- 允许在容器边界或局部标识上加入少量题签式表达
- 不用古典化措辞替代错误或状态说明

## 8. 动效策略

- 使用克制过渡，常规时长控制在 150ms 至 250ms
- 保留已有路由切换与弹层动效的稳定性
- 增加少量淡入、浮层、标题显现等强化质感的过渡

明确避免：

- 大范围弹跳、旋转、夸张位移
- 为炫技而牺牲响应速度或阅读稳定性

## 9. 响应式与可访问性

- 保持现有桌面与移动端结构兼容
- 登录页在窄屏下仍应维持明确的品牌层级与表单顺序
- 控件保持足够点击面积
- 对比度不因纹理和装饰而下降
- 兼容 `prefers-reduced-motion`

## 10. 文案替换原则

统一替换的品牌位：

- 页面标题中的系统名
- 登录页品牌位
- 侧边栏品牌位
- Hero 或页面眉标中的平台称呼

保留业务直白的区域：

- 菜单项
- 表单字段名
- 按钮动作名
- API 返回提示相关文案

## 11. 测试与验证

实现后至少验证以下内容：

- 模板渲染正常，品牌名和关键结构仍可通过测试识别
- 登录页字段 ID、按钮逻辑和改密流程未受影响
- 基础布局在桌面和移动宽度下均可使用
- 内页导航、Hero、表格、表单未出现明显回归

测试方式：

- 更新并执行与模板结构相关的 `unittest`
- 必要时增加对新品牌名或新结构的断言

## 12. 成功标准

完成后应达到以下标准：

- 用户第一眼能记住 `兰台清晖`，但不会看不懂系统用途
- 登录页具备正式、稳重、有文化气质的识别度
- 内页不改变业务路径，但整体完成度明显高于普通后台模板
- 改动集中、边界清晰，不引入接口或行为回归
